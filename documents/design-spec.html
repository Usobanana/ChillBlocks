GS1 — Game Spec

Board & Piece Set

盤面とピース形状の基本仕様。SEO攻略記事ではなく、実際に動く2つのオープンソース実装のソースコードを根拠に更新。

調査ソース（実装コード）： RisticDjordje/BlockBlast-Game-AI-Agent（Python/Pygame、RLエージェント学習用の自作再現実装）と tokaa1/blockerino（TypeScript/React Native、Block Blastクローンアプリ）の2つを実際にクローンしてソースを読んだ。どちらも本家の公式コードではなくサードパーティの再現実装だが、SEO攻略記事の推測とは違い「実際に動いて遊べる」実装であり、2つの独立した実装が一致する点は信頼度が高いと判断できる。

盤面

  • 【確認済み・実装コード】8×8固定グリッド。両実装ともCoreモードは8×8で一致（blockerinoはboardLength = gameMode == Chaos ? 10 : 8と、10×10・5ピース手持ちの「Chaos」という別モードも実装していたが、これは本家Block Blastの確認済み仕様ではなくクローン側の独自追加。ChillBlocksのスコープには含めない）。
  • セル状態は「空」「埋まり」の2値。SortGemsのような目標色（goalColor）の概念はない。

ピーストレイ

  • 【確認済み・実装コード】常に3ピースを保持。3つ全部を配置し終えたタイミングでまとめて新しい3ピースが配られる（両実装とも「手持ち配列が全部nullになったら一括生成」というロジックで一致）。
  • 【確認済み・実装コード】プレイヤー操作としての回転は不可。ただし同じ形状の別向き（L字の4方向など）はそれぞれ別ピースとしてプールに登録されている（ristic実装のFORMS配列で、L字だけで4〜8バリエーションが個別に定義されているのを確認）。以前「想定」としていた点がソースコードで裏付けられた。

ゲームモードの範囲（確認済み・スコープ確認）

確認済み：実際のBlock BlastにはClassic（エンドレス・ハイスコア勝負）とAdventure（レベル制・アプリ版のみ）の2モードが存在する。ChillBlocksが今議論しているのはClassicモード相当（本ドキュメントのGame Play画面もこちら）。Adventure的なレベル制モードは将来拡張のスコープ外として明示しておく。

ピース形状カタログ（実装コードから抽出・確定版）

2つの実装は形状セットが完全一致ではないが、共通する主要形状は次の通り。ChillBlocksではこの合成カタログをそのまま採用してよい。

分類形状イメージ備考
1マス
ristic実装のみ（詰み回避のフォールバック用ピース。blockerinoには無し）
2マス（直線）
両実装で確認。縦横2方向
3マス直線
両実装で確認。縦横2方向。blockerinoでは出現重み4（かなり高頻度）

GS2 — Game Spec

Piece Generation Logic

次に配られる3ピースの生成アルゴリズム。SEO攻略記事は矛盾だらけだったが、実装コード2件からは対照的な2つの現実的な設計が見つかった。

Web検索での限界（前回の記録）

SEO攻略記事は「単純ランダム」「盤面解析して意図的に難しくする」「7回連続で300%補正のピティタイマー」など互いに矛盾し、出典もなく信頼できなかった（詳細は前回の調査ログ参照）。

実装コードで見つかった、対照的な2つの現実解

ristic実装とblockerino実装は、出現ロジックの設計思想が正反対だった。

A: ristic実装（保証付き生成）B: blockerino実装（重み付きランダム）
方式盤面状態を見て、3つとも必ずどこかに置けることを保証してから配る（置けない形状は候補から除外し、置ける形状が見つかるまで探索）盤面は一切見ない。形状ごとの出現重み（distributionPoints）による単純な重み付きランダム抽選を3回行うだけ
盤面が詰む可能性READMEに明記: 「the game engine guarantees that at least one of the three available shapes can always be placed」＝設計上、詰みは起こらない盤面を見ないため理論上「3つとも置けない」詰み状態が起こり得る
本家との整合性前回のWeb調査で見た「本家は時々詰む（no-solution状態が実在する）」という証言と矛盾する「本家は時々詰む」という証言と整合する
ChillBlocksでの採用方針（推奨）： B（blockerino方式の重み付きランダム）を採用する。理由は、前回のWeb調査で確認できた「本家は理不尽に感じるほど難しい組み合わせが来ることがあり、実際に詰むことがある」という実態と、Aの「詰み保証」設計は矛盾するため。Aは著者自身のRLエージェント学習用の独自改良であり、本家の再現を意図したものではない（README内でも本家準拠を主張していない）。Bのほうが「本家らしいランダムさ・理不尽さ」を再現しやすい。

採用する重み付け（blockerino実装からそのまま移植）

形状分類出現重み備考
2×2 正方形6最も出やすい
3マス直線（1×3/3×1）4
3×3 正方形3
L字（8バリエーション）2 each
2マス・4マス直線2 each
T字/三角形（4バリエーション）1.5 each
S字/Z字（4バリエーション）1 each最も出にくい

実装は「全ピースの重み合計に対する乱数位置」で1つ選ぶルーレット選択（blockerinoのgetRandomPiece()と同じロジック）を3回繰り返すだけで良い。盤面状態は見ない。

TBD：上記2実装はいずれも本家の非公開コードではなくサードパーティ再現であり、本家が本当に「盤面を一切見ない重み付きランダム」なのか、あるいはDeconstructor of Funが示唆したような何らかの難易度連動要素を持つのかは、依然として確証がない。まずはB方式で実装し、プレイフィールに不満（理不尽すぎる／簡単すぎる）が出たら、ristic方式のような緩い「置ける保証」を部分的に混ぜる調整を検討する、という段階的な進め方を推奨する。
GS3 — Game Spec

Scoring & Combo

得点計算とコンボ／連鎖ボーナスの仕様。実装コード2件から具体的な式が見つかり、コンボが途切れる条件は2つの独立実装が一致した。

「コンボ」と「ストリーク」は別概念（Web調査時点で確認済み・据え置き）

  • コンボ = 1回の配置で複数ライン（行・列）を同時に消すこと。
  • ストリーク = 連続する手でライン消しを続けること。

ただし実装コード2件を見ると、この2つを1本のcomboカウンターで表現している（消すたびに加算、消せない手が続くとリセット）のが実態に近く、「別々の変数」として厳密に分ける必要は薄いかもしれない。下記の2実装もどちらもcombo変数1本で両方の概念を兼ねている。

実装コードで見つかった、2つの具体的な計算式

A: ristic実装B: blockerino実装
配置しただけの加点なし（ライン消し時のみ加点）置いたピースのマス数だけ加点
ライン消しの基礎計算bonus = 消したライン数 × 10 × (combo数+1)score += 消したライン数 × 盤面サイズ(8) × (combo数/2) × 置いたピースのマス数
複数ライン同時消しの補正3ライン以上なら bonus ×= (ライン数−1)上の式に自然に内包（linesBrokenが式に直接掛かる）
全消し（All Clear）ボーナス+300固定なし（未実装）
comboの増分消したライン数ぶん加算消したライン数ぶん加算

【確認済み・2実装が一致】コンボが途切れる条件

2つの独立した実装が一致：「ライン消しを伴わない配置が3回連続したらコンボを0にリセットする」という条件が、ristic（MAX_COMBO_STREAK = 3）とblockerino（lastBrokenLine >= handSize、handSize=3）の両方でまったく同じ「3」という値で実装されていた。手持ちピース数（3）と揃えているのは自然な設計であり、ChillBlocksでも同じ「3回ライン消し無しでコンボリセット」を採用してよい確度が高い。

ChillBlocksでの採用方針（推奨）

A（ristic方式）を初期実装のベースに採用する。理由: 配置マス数に依存せず「10点×ライン数×(combo+1)」という単純な式で調整しやすく、all-clearボーナスもプレイヤーへの分かりやすい報酬になる。Bの「盤面サイズ×マス数」を絡めた式は数値が急激に大きくなりやすく、初期のバランス調整がしにくい。

イベントChillBlocks初期値（ristic方式を採用）
ライン消し（1本）10 × (combo数+1)
複数ライン同時消し（3本以上）上の値 ×(ライン数−1)
全消しボーナス+300
コンボリセット条件ライン消し無しの配置が3回連続
TBD：これも本家の公式数値ではなくサードパーティ実装からの拝借。実際にプレイしてみてスコアの伸び方が気持ちよいかどうかは調整が必要（特に+300の全消しボーナスの重み付けは実プレイでチューニング推奨）。
GS4 — Game Spec

Monetization（広告）

「1プレイ終わるごとに広告」案を、離脱率とのバランスを見て具体化した仕様。

実機（本家Block Blast）の確認済み事実

確認済み（ゲーム業界分析メディア Deconstructor of Fun）：本家Block Blastは課金アイテム（IAP）を一切持たず、広告収益のみで成立している。同ジャンルでは珍しい設計だが、これで十分ビジネスとして成立することが実例として確認できた。ChillBlocksの初期スコープでも、IAP・ストア機能は作らず広告のみに絞ってよい根拠になる。

基本方針

  • インタースティシャル（強制広告）: Game Over直後に表示。ただし頻度キャップを設け、毎回ではなく例えば「直近の表示から一定回数のGame Overを空ける」形にする（初心者の即時離脱を防ぐ）。
  • リワード広告（任意・プレイヤー起点）: Game Over画面の Continue ボタン。1プレイにつき1回まで、視聴で復活。
  • バナーは使わない（決定）。常時表示のバナーはLo-Fi Chillな画面の余白・雰囲気を常時削るため、GS5のArt Directionと矛盾する。インタースティシャル＋リワードの2枠のみに絞る。

実装アーキテクチャ：SortGemsのAdManagerを移植

ゼロから設計するのではなく、SortGemsで既に動いている SortGemsUnity/Assets/Scripts/Ads/AdManager.cs（Google AdMob Unity Plugin採用、シングルトン、Editor/非モバイルでは自動モック化）をほぼそのまま移植する。広告SDKまわりは枯れた実装を使い回し、ChillBlocks側で新規に書くのはトリガーの呼び出し箇所だけに絞る。

SortGemsでの実装ChillBlocksでの扱い
AdManager（シングルトン、AdMob初期化、Editor時は自動ダミー広告）そのまま移植。変更不要。
OnStageCleared()（ステージクリア毎にNカウント、閾値到達でインタースティシャル表示）OnGameOver()に改名し、ステージクリアの代わりにGame Overのたびに呼ぶ（ロジックは同一、呼び出しタイミングだけ変更）。
ShowRewardedAd(Action onRewardEarned, Action onFallback)そのまま流用。Game Overの Continue ボタンから呼び、onRewardEarnedで復活処理、onFallback（広告未準備時）でボタンをUnavailable表示に切替。
IsRewardedReady / IsInterstitialReady（public bool）そのまま流用。Continueボタンの表示状態（下記）を毎フレーム／表示直前にこれで判定。
ShowBanner() / HideBanner()呼び出さない。ChillBlocksはバナー不使用の方針（上記）のため未使用のまま残しておいてよい。

GS5 — Game Spec

Art & Audio Direction

Lo-Fi Chillな世界観の定義。SortGemsの「Cozy Lo-Fi Pixel Palette」の方向性を継承しつつ、ブロックパズル向けに簡略化する。

ビジュアル

  • タイトル背景・ゲームプレイ背景はComfyUIで生成した静止画1〜2枚（アニメーションなし）。この2枚だけで世界観の大半を作る＝アセット制作コストを最小化する狙い。
  • ブロック本体の配色も、彩度の高い原色ではなく温かみのあるパステル系（SortGems Visual Design Specの方向性を流用）に寄せる。プレイ中は盤面がブロックで埋まるため、背景以上にブロックの配色が体感の雰囲気を左右する点に注意。
  • 枠線は太めで丸みを持たせ、影は黒ではなく紫がかったグレー／ブラウン系に統一（SortGems Do/Don'tルールを踏襲）。

オーディオ

  • BGM: Lo-Fi Hip-Hop調のループ（環境音・雨音レイヤーなども検討）。

差別化の要点：ジャンル自体はBlock Blast系のトレースだが、見た目＋音の両方をLo-Fi Chillに統一することが差別化の軸。背景だけでなくブロックの配色・SEのトーンまで一貫させないと、プレイ中に元のゲームと変わらない印象になってしまう点に注意（過去の相談で指摘済み）。
